# CMS Memory Core — Ver1.0 仕様書（ローカルファースト版）

本書は、CMS Memory Core を「サーバー保存型（SQLite）」から
「ローカルファースト（IndexedDB）」へ移行するにあたっての正式仕様を定めるものです。
**本書は仕様の確定のみを目的とし、この時点では実装を行いません。**

---

## 1. 目的・背景

現行のSQLiteサーバー保存型から、以下を目的としてローカルファースト構成へ移行する。

- **ランニングコスト削減**：サーバー側にデータベースを持たないため、ホスティング費用・運用負荷を最小化できる
- **データをユーザーが所有する設計**：メモは常にユーザーのブラウザ内に存在し、アプリ提供者はデータを一切保持しない
- **プライバシー性向上**：メモ内容がネットワークを経由しないため、通信経路・サーバー保管に伴う漏えいリスクが構造的に存在しない
- **技術記事としての差別化**：「ローカルファーストな個人用メモリーシステム」という設計思想そのものを発信価値とする

## 2. 設計原則

1. **ローカルファースト**：メモ本文・タグ・評価等のユーザーデータは、ブラウザのIndexedDBにのみ保存する。サーバーへの送信・保存は一切行わない。
2. **ユーザーがデータを所有する**：JSONバックアップ（エクスポート／インポート）を正式機能とし、アプリに依存せずデータを取り出せる状態を常に保証する。
3. **追加型の拡張性**：データ構造・バックアップファイル形式のいずれも、将来の機能追加によって既存データを破壊しない設計とする（SQLite時代の「ALTER TABLE以外禁止」という方針を、ローカルファースト構成に合わせて引き継ぐ）。
4. **シンプルさ優先**：機能追加よりも「長期間・継続して使えること」を優先する。
5. **段階的な将来対応**：PWA化・PCアプリ化は本仕様の対象外だが、それらへの移行を妨げない設計にする。

## 3. 全体構成

```
[ブラウザ]
  ├─ UI（React / Next.jsで配信されるクライアントコード）
  └─ IndexedDB（データの唯一の保存先）

サーバー（Next.jsホスティング）は静的なUI配信のみを担い、
メモデータの保存・処理には一切関与しない。
```

- サーバー側API（`/api/notes` 等）およびサーバー側データベース（`cms.db` / better-sqlite3）は本構成には存在しない。
- 将来的にNext.jsを静的書き出し（`output: 'export'`）に変更することも可能な構成とする（本仕様では必須としない）。

## 4. データモデル

### 4.1 Note（メモ）

| フィールド名 | 型 | 必須 | 既定値 | 説明 |
|---|---|---|---|---|
| `id` | string（UUID v4） | ○ | - | `crypto.randomUUID()` で生成。主キー |
| `title` | string | ○ | - | タイトル |
| `body` | string | ○ | - | 本文（全文、要約・切り取りをしない） |
| `userTags` | string[] | ○ | `[]` | ユーザーが自由に追加するタグ |
| `futureValue` | `"高" \| "中" \| "低" \| "未評価"` | ○ | `"未評価"` | 「あとで役立つ」評価。ユーザー自身が判断する |
| `createdAt` | string（ISO 8601） | ○ | - | 作成日時 |
| `updatedAt` | string（ISO 8601） | ○ | - | 最終更新日時 |
| `ai` | object（省略可） | - | `undefined` | **将来のAI機能拡張用の予約フィールド**。Ver1.0では一切読み書きしない |

`ai` フィールドの想定構造（将来用・Ver1.0では未使用）：

```ts
ai?: {
  summary?: string;
  tags?: string[];
  type?: string[];
  titleSuggestion?: string;
  status?: string;
}
```

> SQLite時代は「未使用列をあらかじめ`ALTER TABLE`で追加しておく」方式だったが、
> IndexedDB＋JSONオブジェクトの構成ではその必要がない。
> 新フィールドは既存レコードに存在しなくても型定義上オプショナルとして扱えばよく、
> 「将来追加されるフィールドが欠けていても壊れない」ことを設計の前提とする。

### 4.2 IndexedDBの物理設計

| 項目 | 内容 |
|---|---|
| データベース名 | `cms-memory-core` |
| データベースバージョン | `1`（整数。IndexedDB標準のバージョニング機構） |
| ObjectStore | `notes`（keyPath: `id`） |
| インデックス | `createdAt`、`updatedAt`、`futureValue`（いずれも非ユニーク） |

- 将来的にstoreを追加する場合（例：`embeddings`、タグのマスタ管理等）は、IndexedDBのバージョンを上げ、`onupgradeneeded`内で追加のみを行う。既存storeの削除・再作成は行わない。

## 5. 機能仕様（Ver1.0で提供する機能）

### 5.1 メモの作成
- 入力項目：タイトル（必須）、本文（必須）、ユーザータグ（任意・複数可）、あとで役立つ評価（任意・既定は「未評価」）
- 保存後は一覧画面（新しい順）へ遷移する

### 5.2 一覧
- 並び替え：新しい順／古い順／あとで役立つ順（高→中→低→未評価）
- 表示項目：タイトル・ユーザータグ・あとで役立つ評価・作成日時のみ（本文の抜粋表示はしない）
- カード上から直接削除可能（確認あり）

### 5.3 検索
- キーワード未入力時：最近のメモを新しい順で一覧表示
- キーワード入力時：タイトル・本文・ユーザータグを対象に絞り込み
- 全角/半角・大文字/小文字の表記ゆれを吸収する（Unicode正規化＋小文字化をJS側で実施）
- 並び替え：新しい順／古い順／あとで役立つ順／更新順

### 5.4 詳細画面
- 表示順：タイトル → 本文 → あとで役立つ → ユーザータグ → メモ管理
- あとで役立つ評価はいつでも変更可能
- ユーザータグは追加（複数タグ入力対応）・削除が可能

### 5.5 タグ履歴
- 直近使用した最大20件のタグを、記録画面の入力欄付近に候補表示する
- クリックで入力欄に追加（入力補助）
- 詳細画面には表示しない（役割を分離）

### 5.6 削除
- 一覧・詳細のいずれからも削除可能
- 削除前に必ず確認画面を表示する
- 対象1件のみを削除し、他のメモには影響しない

### 5.7 バックアップ（エクスポート）
- 実行前に確認画面を表示し、保存内容と用途を説明する
- IndexedDB内の全メモをJSON化し、ファイルとしてダウンロードする（形式は6章に定める）
- 完了後、件数・ファイル名・「外部には送信されないこと」を通知する

### 5.8 復元（インポート）
- JSONファイルを選択すると、まず現在のデータと比較し、件数（追加件数・変更件数）のみを表示する（全件一覧は表示しない）
- ユーザーが確認したうえで実行を選択した場合のみ反映する
- 反映ルールは「追加」「更新」のみで、**現在存在するメモを削除することは一切ない**
- 大量データでも実用的に動作する設計とする（全件を一括読み込みし、IDをキーに突き合わせる方式）

## 6. JSONバックアップ形式仕様

### 6.1 トップレベル構造

```json
{
  "formatVersion": 1,
  "app": "cms-memory-core",
  "exportedAt": "2026-07-08T12:00:00.000Z",
  "noteCount": 12,
  "notes": [
    {
      "id": "5f2b6a...",
      "title": "読書メモ",
      "body": "...",
      "userTags": ["読書", "学習"],
      "futureValue": "高",
      "createdAt": "2026-07-01T09:00:00.000Z",
      "updatedAt": "2026-07-02T10:00:00.000Z"
    }
  ]
}
```

### 6.2 バージョニング方針

- `formatVersion` は整数。バックアップファイルの構造そのものを識別する（アプリのバージョンとは独立）。
- **後方互換の範囲内（フィールド追加等）は `formatVersion` を上げない**。読み込み側は未知のフィールドを無視し、欠けているフィールドには既定値を補う。
- **既存フィールドの意味変更・削除など破壊的変更が必要な場合のみ** `formatVersion` をインクリメントし、インポート時に「旧バージョン→現行バージョン」への変換関数を通す。
- 現行のSQLite版が出力するバックアップJSON（`formatVersion`フィールドを持たない）は `formatVersion: 0` とみなし、自動変換して読み込めるようにする（既存ユーザーの資産を失わないため）。
- **未知（未来）の`formatVersion`を読み込んだ場合**：このアプリが対応する`formatVersion`（現行は`1`）と一致しない場合は、変換を試みず「対応していない形式です」という明確なエラーで復元を中止する。中途半端な変換によるデータ破損より、安全に失敗することを優先する。
- **`app`フィールドの検証**：`formatVersion >= 1`のファイルは`app`フィールドが`"cms-memory-core"`と一致することを必須とし、一致しない場合は復元を中止する（無関係なJSONファイルを誤って読み込むことを防ぐ）。`formatVersion: 0`（レガシー形式）にはこの検証を適用しない。
- **`noteCount`は参考値**：件数の正本は常に`notes.length`であり、`noteCount`はプレビュー表示用の付随情報に過ぎない。実装は件数判定に`noteCount`を用いてはならない。
- **未知フィールドの保持**：`Note`オブジェクトに含まれる未知のフィールド（将来の`ai`拡張など、この時点のアプリが読み書きしないもの）は、復元時に削除せずそのまま保持する。IndexedDBはスキーマレスであるため、既知フィールドの検証・既定値補完のみを行い、それ以外のキーは素通しすることで実現する（`formatVersion: 0`のレガシー形式にはこの概念がないため対象外）。

### 6.3 復元時の比較・反映ルール（5.8の再掲・詳細）

- 現在のIndexedDBの全件と、インポートするJSONの全件を`id`で突き合わせる
- 「内容が同一」の判定対象は `title` / `body` / `userTags` / `futureValue` の4項目のみとする。`createdAt` / `updatedAt` はタイムスタンプであり「内容」には含めない
- IDが一致し内容が同一 → 「変更なし」としてカウントのみ
- IDが一致し内容が異なる → 「更新」対象（**バックアップの内容を正として無条件に上書きする**。`updatedAt`の新旧比較（いわゆる「新しい方を優先」）は行わない。復元はユーザーが内容を確認した上で明示的に実行する手動操作であり、本アプリは単一デバイス運用を前提としているため、この単純なルールを採用する）
- バックアップにのみ存在するID → 「追加」対象（バックアップの`createdAt`/`updatedAt`をそのまま保持する）
- **現在のIndexedDBにのみ存在するメモ（バックアップに含まれないメモ）は、削除せずそのまま残す**

## 7. 将来拡張の考慮点（Ver1.0の対象外・設計上ブロックしない事項）

| 項目 | Ver1.0での扱い |
|---|---|
| PWA化（Service Worker・manifest.json） | 対象外。データ層（IndexedDB）とは独立しているため、後から追加してもデータ設計への影響はない |
| PCアプリ化（Electron/Tauri等） | 対象外。ラップする実行環境によってIndexedDBの永続化挙動が異なる可能性があるため、着手時に個別検証が必要 |
| AI機能の再導入 | `Note.ai`フィールドを予約済み。Ver1.0では一切読み書きしない |
| 複数端末・複数ブラウザ間の同期 | **Ver1.0の対象外**。IndexedDBは端末・ブラウザごとに独立したストレージであるため、同期には別途クラウド同期層の設計が必要になる。現時点ではJSONバックアップの手動移行が唯一の手段となる、という制約を既知の事実として明記する |

## 8. 非機能要件

- **プライバシー**：メモ本文を含むいかなるユーザーデータも、外部サーバー・第三者サービスへ送信しない
- **ブラウザ対応**：IndexedDBをサポートする主要モダンブラウザ（Chrome / Edge / Firefox / Safari）を対象とする。SafariのプライベートブラウジングモードではIndexedDBの利用に制限がある場合があることに留意する
- **パフォーマンス**：個人利用を想定したデータ量（数千件程度）において、一覧表示・検索・バックアップ／復元が実用的な速度で動作すること
